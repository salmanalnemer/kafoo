# تقرير النسخة الأمنية النظيفة — نظام كفو

**التاريخ:** 17 يوليو 2026  
**اسم الحزمة:** `Kafo_Secure_Clean_2026-07-17.zip`

## الحالة

تم إعداد نسخة مصدر نظيفة ومشددة أمنيًا من المشروع المرفق. لا تحتوي الحزمة على قاعدة بيانات، بيانات مستخدمين، ملفات مرفوعة، أسرار تشغيل، نسخ احتياطية، أو مخرجات بناء `bin/obj`.

الإصلاحات أدناه هي إصلاحات على مستوى التطبيق والمصدر. إعداد الخادم، TLS، الجدار الناري، قاعدة البيانات الفعلية، ClamAV، النسخ الاحتياطية، والمراقبة التشغيلية يجب تنفيذها وفق قائمة النشر المرفقة داخل المشروع.

## الإصلاحات الأمنية المطبقة

### 1. حسابات الدخول وكلمات المرور

- إزالة حساب المدير وكلمة المرور الافتراضية الثابتة من المصدر.
- إنشاء المدير الأول فقط من متغيرات بيئة مؤقتة، مع التحقق من قوة كلمة المرور.
- استبدال التجزئة اليدوية بـ `ASP.NET Core Identity PasswordHasher` وإعداد 600,000 دورة.
- دعم التحقق من التجزئة القديمة وترقيتها تلقائيًا عند أول دخول صحيح.
- تطبيق سياسة كلمة مرور بطول 12–128 حرفًا مع متطلبات التعقيد ومنع كلمات المرور الشائعة والمتسلسلة.
- تطبيق قفل مؤقت للحساب بعد تكرار المحاولات الفاشلة.
- إضافة تأخير وحدود طلبات على تسجيل الدخول وواجهات OTP والنماذج العامة.
- منع كشف وجود الحساب من خلال رسائل الدخول.

### 2. المصادقة متعددة العوامل وOTP

- تطبيق OTP على تسجيل دخول الإدارة وبوابات الداعمين والجهات.
- تخزين تحديات OTP في قاعدة البيانات بدل ذاكرة التطبيق.
- حماية الرمز باستخدام ASP.NET Core Data Protection بدل تخزين الرمز الصريح.
- تقييد الإرسال حسب البريد وعنوان IP.
- تحديد مدة صلاحية، وعدد محاولات تحقق، وعدد مرات إرسال.
- إبطال التحديات السابقة واستخدام الرمز مرة واحدة فقط.
- استخدام مقارنة ثابتة الزمن عند فحص الرمز.

### 3. إعداد واستعادة كلمات المرور

- إلغاء إرسال كلمات المرور المؤقتة عبر البريد.
- إنشاء رابط HTTPS عشوائي، أحادي الاستخدام، ومحدود الصلاحية لإعداد كلمة المرور.
- تخزين تجزئة الرمز فقط في قاعدة البيانات.
- إبطال الرموز السابقة عند إصدار رمز جديد.
- تدوير `SecurityStamp` بعد تغيير كلمة المرور أو البريد أو حالة الحساب.

### 4. حماية الجلسات وملفات الارتباط

- إضافة `SecurityStamp` لكل حساب والتحقق منه في كل طلب مصادق عليه.
- إبطال الجلسات السابقة عند تغيير كلمة المرور أو البريد أو تعطيل الحساب.
- تقليل مدة الجلسة الافتراضية إلى ساعتين.
- تعطيل `Sliding Expiration`.
- ضبط ملفات الارتباط على `HttpOnly` و`Secure` في الإنتاج و`SameSite=Strict`.
- تحويل تسجيل الخروج إلى طلب POST محمي بـ Anti-forgery.
- منع الحساب الذي يجب عليه تغيير كلمة المرور من استخدام بقية الصفحات قبل تنفيذ التغيير.

### 5. رفع الملفات وتخزينها

- نقل الملفات الخاصة خارج `wwwroot`، وتشمل السير الذاتية، مرفقات التواصل والتقييم، مرفقات طلبات التوظيف، وتقارير الداعمين.
- إنشاء Controller آمن للتنزيل يتحقق من تسجيل الدخول والدور وملكية الملف وصلاحية الصفحة.
- حظر المسارات القديمة للملفات الخاصة داخل `/uploads`.
- إضافة خدمة ترحيل تلقائي للملفات الخاصة القديمة من المسار العام إلى التخزين المحمي عند ترقية بيئة قائمة.
- تقييد مجلدات الرفع بقائمة سماح ثابتة.
- تخصيص الامتدادات والأحجام المقبولة لكل نوع استخدام.
- التحقق من اسم الملف والامتداد وMIME والتوقيع الحقيقي `Magic Bytes`.
- إنشاء أسماء تخزين عشوائية ومنع Path Traversal.
- فحص الملفات بواسطة ClamAV قبل الحفظ.
- ضبط فحص البرمجيات الخبيثة على `Fail-Closed` في الإنتاج.
- حذف الملف الجديد عند فشل حفظ التغييرات، وعدم حذف الملف القديم قبل نجاح الرفع البديل.

### 6. XSS وحماية العرض

- إزالة استخدامات `Html.Raw` من المحتوى العام.
- إزالة تعيينات `innerHTML` من كود التطبيق واستبدالها بواجهات DOM آمنة.
- إزالة `onclick` و`onchange` و`onsubmit` و`onerror` المضمنة من صفحات Razor.
- إضافة Tag Helper لإنشاء `nonce` لكل استجابة وتطبيقه على السكربتات.
- تطبيق CSP تمنع `script-src-attr` وتمنع السكربتات المضمنة غير المصرح بها.
- إضافة تحقق مخصص للروابط الخارجية؛ HTTPS فقط، أو مسار محلي آمن حسب نوع الحقل.
- إضافة `rel="noopener noreferrer"` لجميع الروابط التي تستخدم `target="_blank"`.

### 7. CSRF والترويسات الأمنية

- تفعيل `AutoValidateAntiforgeryToken` على عمليات تغيير الحالة.
- تطبيق Content Security Policy.
- تطبيق HSTS في الإنتاج.
- إضافة `X-Content-Type-Options: nosniff`.
- إضافة حماية الإطارات عبر CSP و`X-Frame-Options`.
- إضافة `Referrer-Policy` و`Permissions-Policy` وCOOP وCORP.
- منع التخزين المؤقت للملفات الخاصة.

### 8. Host Header وReverse Proxy وHTTPS

- إزالة `AllowedHosts: "*"` واستبداله بنطاقات محددة قابلة للتعديل.
- إضافة `Security:PublicBaseUrl` وعدم توليد روابط البريد من `Request.Host`.
- قبول روابط إعداد الحساب عبر HTTPS فقط.
- إعداد `ForwardedHeaders` مع `ForwardLimit` وقائمة `KnownProxies`.
- إضافة إعدادات Nginx وsystemd ومتغيرات البيئة كنماذج نشر.

### 9. التدقيق والسجلات والمراقبة

- إنشاء سجل تدقيق أمني في قاعدة البيانات للعمليات الحساسة، تسجيل الدخول، تسجيل الخروج، والتنزيلات الخاصة.
- تسجيل عنوان IP، User-Agent، المسار، المستخدم، النتيجة، ومعرف الارتباط.
- إضافة سجلات ملفات يومية بواسطة Serilog مع مدة احتفاظ محددة.
- إضافة خدمة تنظيف دورية لتحديات OTP والرموز والسجلات القديمة.
- فصل الأخطاء التشغيلية عن سجل التدقيق الأمني.

### 10. سلسلة التوريد والفحص المستمر

- تثبيت SDK عبر `global.json` على `10.0.302`.
- تحديث حزم Microsoft الأساسية إلى `10.0.10`.
- تفعيل NuGet Audit داخل `Directory.Build.props`.
- إضافة فحص ساكن مستقل عن SDK في `scripts/security-static-check.py`.
- إضافة أوامر فحص Windows وLinux للبناء وفحص الحزم الضعيفة والمتقادمة.
- إضافة GitHub Actions للبناء والفحص الأمني عند Push وPull Request.

## نتائج التحقق المنفذ

- نجح الفحص الأمني الساكن المرفق: `SECURITY STATIC CHECK: PASSED`.
- نجح تحليل جميع ملفات JSON.
- نجح تحليل ملفات المشروع XML.
- نجح فحص البنية والأقواس لـ 368 ملف C#.
- لم توجد كلمات مرور افتراضية ثابتة أو `Html.Raw` أو `innerHTML` في كود التطبيق.
- لم توجد معالجات أحداث JavaScript مضمنة في صفحات Razor.
- جميع روابط `target="_blank"` تحتوي `noopener noreferrer`.
- لم توجد قواعد بيانات، مفاتيح، شهادات، ملفات نسخ احتياطية، أو ملفات مستخدمين في الحزمة.
- تمت إعادة فتح ZIP وفحص جميع أسمائه ومحتوياته بعد إنشائه.

## قيد التحقق

لم يتم تنفيذ `dotnet restore` أو `dotnet build` داخل بيئة إعداد الحزمة لعدم توفر .NET SDK وإتاحة تنزيله في بيئة التنفيذ. لذلك لا يُعد الفحص الساكن بديلًا عن البناء الفعلي. الحزمة تتضمن سكربتات جاهزة لتنفيذ البناء وفحص NuGet قبل النشر:

```powershell
.\scripts\security-check.ps1
```

أو:

```bash
./scripts/security-check.sh
```

## متطلبات إلزامية قبل الإنتاج

1. ضبط النطاق الفعلي في `AllowedHosts` و`Security__PublicBaseUrl`.
2. وضع كلمة مرور SMTP في Secret Store أو متغير بيئة فقط.
3. ضبط مسارات قاعدة البيانات والرفع الخاص ومفاتيح Data Protection خارج مجلد الإصدار.
4. تشغيل ClamAV وتحديث تواقيعه واختبار رفض ملف EICAR في بيئة معزولة.
5. ضبط Nginx وTLS وKnown Proxies وعدم كشف منفذ Kestrel مباشرة.
6. تشغيل سكربت الفحص والبناء ومعالجة أي خطأ تجميع أو حزمة ضعيفة.
7. تطبيق الترحيلات على نسخة Staging واختبار جميع رحلات الدخول والOTP والرفع والتنزيل.
8. تنفيذ DAST واختبار اختراق مصرح به على Staging قبل اعتماد الإنتاج.
9. تشفير النسخ الاحتياطية واختبار الاستعادة ووضع سياسة احتفاظ.
10. مراقبة السجلات والتنبيه على محاولات الدخول الفاشلة والرفع المرفوض والنشاط غير المعتاد.
